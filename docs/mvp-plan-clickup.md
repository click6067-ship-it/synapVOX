# 회의 지식 파이프라인 MVP 구현 계획

회의 지식 파이프라인 MVP 구현 계획

1. 문서 위상 (2차 설계서와의 관계)

본 문서는 \*\*1차 범위(녹음·전사·단일 세션 요약)\*\*의 구현 계획이다. 그 위에 얹히는 \*\*세션 간 지식 연결 계층(Graph RAG)\*\*은 별도 상세 설계서 "회의 맥락 RAG 시스템 — 상세 설계서 v1.1"(2차 기능)에서 다룬다. 본 MVP는 그 2차 설계가 전제하는 토대이므로, 다음 두 가지를 의식하고 구현한다.

2차가 정의한 차별화 두 축을 MVP가 막지 않도록 설계한다.
축 ① 세션 간 연결 — Topic Thread Graph, SUPERSEDES 이력, 하이브리드 검색. MVP의 "주제 태그"와 "결정/액션 추출"이 이 축의 씨앗이다.
축 ② 사전 자료 기반 동적 STT 보정 — 자료 문서 시딩 + Doc-RAG 주입 + 회의 유래 용어 자동 축적. MVP의 "PPT 텍스트를 LLM 컨텍스트로 투입"이 이 축의 원형이다.
MVP 산출물의 스키마가 2차로 매끄럽게 승격되도록 필드 이름과 ID 규칙을 미리 맞춘다. (아래 3-4절 "2차 연결을 위한 씨앗 필드")

MVP 원칙: 전처리는 코드로, 판단만 LLM으로. Graph DB·오케스트레이션 프레임워크·청킹 로직은 MVP에서 제외하고, 일반 DB + 단일 LLM 호출로 시작한다. 단, 2차에서 도입될 개념(canonical entity ID, Segment, TopicThread, Doc store)의 자리를 MVP 스키마에 미리 비워둔다.

1. 목표

Input: 회의 녹음본(오디오), 회의 PPT·문서

Output:
녹음본 단위: ① 요약본 ② 북마크(타임스탬프 기반 핵심 구간)
프로젝트 단위: ① 시간별 정리 ② 맥락(주제)별 정리

2차 대응: 녹음본 단위 = Meeting(물리적 시간 축), 맥락별 정리 = TopicThread(주제 축)의 원형. MVP에서는 "주제 태그"라는 평면 구조로 시작하고, 2차에서 이를 회의를 가로지르는 Thread 그래프로 승격한다.

1. 아키텍처 (3단계 직선 파이프라인)

\[녹음본 / PPT\]
↓
① 전처리 (비-LLM)
*   STT API (화자분리 내장, 타임스탬프 세그먼트)
*   PPT/문서 텍스트 추출 (python-pptx 등)
*   (2차 예고) 이 PPT 텍스트가 축 ②의 "자료 시딩 + Doc store"로 확장됨

↓

② LLM 호출 1회 (녹음본당)

*   요약 + 북마크 + 주제태그 + 결정 + 액션아이템을

구조화 JSON 하나로 추출

*   (2차 예고) 이 단일 호출이 축 ①의 Segmentation → 관계 판정

다단계 파이프라인으로 분해됨

↓

③ DB 저장 (SQLite → 필요 시 Postgres)

*   시간별 정리 = 날짜 정렬 쿼리
*   맥락별 정리 = 주제 태그 필터 + 임베딩 검색
*   (2차 예고) 주제 태그가 TopicThread 노드로, 임베딩 검색이

Graph+Vector 하이브리드 검색으로 승격됨

직선 → 그래프 전환의 이음새: MVP는 "1녹음본 = 1 LLM 호출 = 1 요약"의 직선 구조다. 2차는 이 안에서 회의를 여러 Segment로 분할하고 각 Segment를 기존 TopicThread와 관계 판정(CONTINUES/BRANCHED\_FROM/REVIVES/SUPERSEDES/MERGE/NEW\_THREAD)하는 분기 구조로 바뀐다. 따라서 MVP의 "bookmarks\[\].type" 과 "topics\[\]" 를 버리지 않고 승격 가능한 형태로 남긴다(3-4절).

1. 핵심 설계 결정

3-1. 전처리
STT는 화자분리가 내장된 관리형 API 사용 (예: CLOVA Speech). pyannote 직접 구현 금지 → 코드량 대폭 절감
표준 중간 포맷으로 고정:

{
"source": "meeting\_2026-07-10.m4a",
"meeting\_id": "M07",
"date": "2026-07-10",
"mode": "meeting",
"segments": \[
{"id": 12, "speaker": "A", "start": 754.2, "end": 761.8, "text": "..."}
\]
}

PPT는 슬라이드별 텍스트를 추출해 같은 회의의 컨텍스트로 LLM 호출에 함께 투입 (전문용어 오인식 교정 효과)
meeting\_id / mode 필드 추가 (2차 대비): 2차의 Meeting 노드와 회의/강의 모드 분기를 위해 미리 자리를 잡는다. MVP에서는 mode를 항상 "meeting"으로 두고 분기 로직은 넣지 않는다.
(축 ② 예고) 자료 문서의 이중 용도: MVP는 PPT를 "LLM 컨텍스트 1회 투입"으로만 쓰지만, 2차에서 같은 문서가 (a) 세션 전 용어 시딩과 (b) Doc-RAG 문맥 소스로 재사용된다. MVP 단계에서 추출한 PPT 텍스트를 버리지 말고 원본 경로와 함께 저장해 두면 2차 Doc store 인덱싱의 입력이 된다.

3-2. LLM 호출 (녹음본당 1회)
장문 컨텍스트 모델 사용 → 청킹/map-reduce 불필요. 전사 전체 + PPT 텍스트를 한 번에 투입
출력 스키마 (JSON 강제):

{
"summary": "안건별 논의 요약...",
"segments": \[
{"raw\_span": \[10, 24\], "summary": "...", "topics": \["결제 모듈"\],
"key\_entities": \["...", "..."\], "discourse\_signals": \[\]}
\],
"bookmarks": \[
{"segment\_id": 12, "type": "decision|action|topic\_shift|conflict", "label": "..."}
\],
"topics": \["결제 모듈", "일정 조정"\],
"decisions": \[{"statement": "...", "supersedes\_hint": null}\],
"action\_items": \[{"task": "...", "assignee": "...", "due": "..."}\]
}

타임스탬프 환각 방지: LLM은 segment\_id만 선택, 실제 타임스탬프는 코드에서 역참조
엔티티 해소 단순화: 프롬프트에 기존 DB의 주제 태그 목록을 넣고 "기존 태그 재사용 우선, 신규는 최소화" 지시 → 별도 엔티티 해소 파이프라인 대체. (2차 예고) 이 "태그 재사용 우선" 지시가 2차의 사전 기반 3단 매칭(exact → alias → 자모 fuzzy)과 canonical entity ID로 대체된다.
(2차 대비) 얕은 Segmentation 씨앗: 출력에 segments\[\](주제 단위 분할 + raw\_span)를 선택적으로 포함시켜, 2차 P1 Segmentation·P2 관계 판정으로 가는 다리를 놓는다. MVP에서는 이 필드를 요약 보조로만 쓰고 관계 판정은 하지 않는다.
(2차 대비) 결정 이력 힌트: decisions\[\].supersedes\_hint로 "이전 결정을 뒤집는 발화" 후보만 표시. 실제 SUPERSEDES 체인 구축은 2차로 미룬다.
검증 루프: JSON 스키마 검사 + segment\_id 존재 확인 실패 시 에러 피드백과 함께 최대 2회 재시도

3-3. DB (테이블 3개)

시간별 정리: SELECT ... ORDER BY date
맥락별 정리: WHERE topic\_tag = ? + 임베딩 유사도 검색 → 검색 결과를 LLM에 넣어 정리문 생성
entity\_id 컬럼 (2차 브리지): 2차 설계의 핵심 원칙 "엔티티는 공유, 저장소는 분리 — 모든 저장소가 canonical entity ID를 공유"에 대비한 자리. MVP에서는 topic\_tag를 그대로 문자열로 쓰되, 2차 Entity Store 도입 시 이 컬럼에 ENT\_xxxxx를 채워 그래프·벡터·사전을 교차 링킹한다.
doc\_paths / meeting\_id / mode (2차 브리지): 각각 Doc store 인덱싱 입력, Meeting 노드 매핑, 회의/강의 모드 분기의 씨앗.

3-4. 2차 연결을 위한 씨앗 필드 (요약)

MVP 스키마에 미리 심는 필드와, 2차에서 무엇으로 승격되는지의 대응표. "저장은 풍부하게, 주입은 선별적으로" 원칙(2차)에 따라, 지금 당장 안 써도 저장해 둔다.

1. 구현 순서

Step 0 — STT 품질 검증 (최우선, 반나절)
실제 녹음본 1~2개로 STT 정확도 확인. 여기가 무너지면 이후 전부 무의미
(축 ② 관점) 도메인 용어 오인식 목록화: 이때 STT가 틀리는 전문용어를 기록해 두면, 2차 자료 시딩·hotword 주입의 효과 측정(E2E STT 개선율)의 baseline이 된다.

Step 1 — 녹음본 단위 파이프라인 (1주)
STT 연동 → 중간 포맷 변환 → LLM 호출 → JSON 검증 → 로컬 파일 출력([요약.md](http://요약.md) + 북마크 목록)
산출물: 오디오 파일 하나 넣으면 요약본+북마크가 나오는 CLI 스크립트

Step 2 — DB 축적 + 프로젝트 단위 조회 (1주)
SQLite 테이블 3개 생성, 파이프라인 결과 저장
시간별 정리 뷰 + 주제 태그별 정리 뷰 구현
임베딩 검색(items.embedding) 추가 → "X 관련 논의 뭐 있었지?" 질의응답

Step 3 — 품질 개선 + 2차 진입 준비 (이후)
북마크 정확도 튜닝, 액션 아이템 → ClickUp Task 자동 생성 연동 검토
2차 진입 게이트: 아래 5절 "도입 조건"이 충족되고, 씨앗 필드(3-4절)가 실데이터에 안정적으로 채워지면 2차 Graph RAG 설계서의 구현 우선순위 1번(정규화 사전 + Entity Store)으로 넘어간다.

1. MVP에서 의도적으로 제외한 것 (도입 조건 → 2차 설계서 매핑)

각 제외 항목의 "도입 시점"이 실제로 도래하면, 그 구현은 2차 상세 설계서의 해당 절을 따른다.

1. 예상 규모

Python 파일 12개, 200300줄 수준

외부 의존성: STT API, LLM API, sqlite3, python-pptx, 임베딩 API

(2차 진입 시 추가 예상: Neo4j, LangGraph, 자모 분해 라이브러리(hgtk), Doc store용 벡터 인덱스)

2. 4인 역할 분담

파이프라인의 3단계 + 통합 축으로 나눈다. 각 역할 사이의 계약(contract)은 3개의 스키마(중간 포맷 JSON, LLM 출력 JSON, DB 테이블 스키마)이며, 이 3개를 첫 주에 함께 확정하면 이후 병렬 작업이 가능하다. 3개 스키마를 확정할 때 3-4절의 씨앗 필드를 포함시켜, 2차 진입 시 스키마 재설계 비용을 줄인다.

담당 A — 데이터 파이프라인 / 전처리 : 현우님
Baseline: STT API 선정·연동, Step 0 품질 검증 주도, 화자분리 결과 확인, PPT/문서 텍스트 추출, 중간 포맷 JSON 변환기 구현
고도화: 화자 이름 매핑(A/B → 실명), 잡음·겹침 발화 대응, 다양한 입력 포맷 확장(화상회의 자동 녹화 연동 등), 프로젝트 단위 질의응답(RAG) 구현
2차 연계: 축 ② 자료 시딩 파이프라인(2.4절) + Doc store 인덱싱, STT hotword 실시간 주입(2차 우선순위 7번). PPT 텍스트 저장 구조를 Doc store 입력으로 확장.
소유 스키마: 중간 포맷 JSON

담당 B — LLM / 프롬프트 & 품질 : 도윤님
Baseline: 출력 JSON 스키마 설계, 요약·북마크·추출 프롬프트 작성, 스키마 검증 + 재시도 루프 구현, 평가용 정답셋(회의 3~5개 수동 라벨링) 구축
고도화: 북마크 정확도 튜닝, 주제 태그 일관성(엔티티 해소) 개선, 결정 변경 추적(SUPERSEDES) 로직, 모델/프롬프트 A/B 평가 체계
2차 연계: 4장 Prompt 설계(P1 Segmentation, P2 관계 판정, P3 Rolling summary, P4 신규 용어 구조화) + 7장 평가 체계(4계층). MVP 정답셋을 2차 골든셋(관계 트리플, mention→ID)으로 확장.
소유 스키마: LLM 출력 JSON

담당 C — 백엔드 / DB & 검색 : 용하님
Baseline: SQLite 테이블 3개 설계·구현, 저장 로직, 시간별·주제별 조회 쿼리, 임베딩 생성·유사도 검색
고도화: Postgres + pgvector 이전, Graph DB 도입 필요성 검증 및 설계(도입 조건 충족 시)
2차 연계: Neo4j 그래프 스키마(3장) + Entity Store SSOT(2.1절) + 하이브리드 검색 View(5.4절). entity\_id 컬럼을 canonical ID로 승격, 벡터·그래프·Doc store 교차 링킹(Cross-store 일관성).
소유 스키마: DB 테이블 스키마

담당 D — 통합 / 제품 (PM 겸) : 도원님
Baseline: E2E 통합(CLI 파이프라인 조립), 실제 회의로 전체 흐름 테스트, "좋은 요약/북마크" 수용 기준 정의, 일정·우선순위 관리
고도화: 액션 아이템 → ClickUp Task 자동 생성 연동, 웹 UI(요약 열람 + 북마크 클릭 시 해당 구간 오디오 재생), 사용자 피드백 루프(북마크 수정 → 프롬프트 개선 반영), front end(ui/ux)
2차 연계: 회의/강의 모드 분기(6장) 제품 정의, REVIVES 기반 "이전 논의 참조 북마크"(축 ① 고유 기능) UX, E2E 시스템 평가(7.4절) 운용.
소유 스키마: (없음 — 통합)

협업 규칙
첫 주: 4인이 함께 3개 스키마 확정 (씨앗 필드 포함) → 이후 각자 병렬 개발
주 1회 통합: 담당 D 주도로 E2E 실행, 실패 지점을 스키마 계약 위반 여부로 판별
스키마 변경은 소유자가 제안하고 영향받는 담당자 합의 후 반영 (버전 명시)

1. Baseline 이후 고도화 로드맵 → 2차 Graph RAG 진입

→ 2차 (Graph RAG) 구현 우선순위 (상세 설계서 8장)

MVP 고도화가 안정화되면 아래 순서로 2차에 진입한다. 각 단계는 앞 단계 평가 지표가 목표치를 충족한 뒤 진행하며, 특히 관계 판정(P2) macro-F1이 0.7 미만이면 이후 그래프가 오염되므로 프롬프트 개선을 우선한다.

정규화 사전 + Entity Store (자모 매칭 포함) — 모든 것의 전제
자료 문서 시딩 + Doc store — 차별화 축 ②, 첫 세션 전 가동
